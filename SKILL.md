# 工单查询

帮助用户查询自己管理范围内的工单，并按需要导出结果。

## 何时使用

除了 description 里列的关键词，以下场景也应触发本skill：
- 用户问"我这边还有多少单没处理完"
- 用户想知道某个客户/某个时间段的工单情况
- 用户要一份工单清单发给别人（即"导出"意图）

如果用户问的是工单的"内容修改""转派""关闭"等操作类需求，这些属于其他skill（工单操作类），本skill只负责"查"，不负责"改"。如果不确定用户是想查还是想改，先按查询处理，返回结果后再询问是否需要操作。

## 前置说明

调用接口前必须明确当前用户身份（user_id / token），所有查询都应限定在该用户有权限查看的数据范围内，不要尝试查询其他用户的工单，也不要在结果里展示不属于当前用户权限范围的字段（如客户联系方式等敏感信息，除非接口本身返回且业务允许展示）。

## 调用流程

1. **确认查询条件**
   与用户对话中提取以下参数，缺失的用默认值，不要为了凑齐条件反复追问用户：
   - 状态（status）：待处理 / 处理中 / 已完成 / 已关闭 / 全部。用户未指定时默认查询"待处理 + 处理中"（即"未完结"的单子）
   - 时间范围（start_time / end_time）：用户未指定时默认最近30天
   - 关键词（keyword）：如客户名、工单标题关键字，用户提到具体名称时才带
   - 分页（page / page_size）：默认 page=1，page_size=20，结果超过一页且用户明显想看全部时才翻页拉取

2. **调用查询接口**
   接口的具体 endpoint、请求方式、参数名、返回字段定义参见 `references/api.md`，调用前先读该文件确认最新的接口签名，不要凭记忆猜字段名。

3. **处理返回结果**
   - 空结果：明确告诉用户"当前条件下没有符合的工单"，可以主动建议放宽条件（比如时间范围）
   - 结果较多（超过20条）：先给用户看数量和整体分布（按状态/紧急程度统计），而不是一股脑列全部，除非用户明确要看完整列表或要导出
   - 接口报错：不要把原始报错堆给用户，转成人话解释，比如权限不足/网络异常/参数错误，并给出下一步建议

4. **导出场景**
   用户提到"导出""发给我""生成清单"时，将查询结果整理成表格（CSV 或用户要求的格式），通过 code execution 生成文件后交付，不要只是把表格贴在对话里当作"导出"。

## 输出格式规范

在对话中展示工单列表时，使用这个结构：

| 工单号 | 标题 | 状态 | 负责人 | 创建时间 |
|---|---|---|---|---|

结尾附一句统计摘要，例如："共12条，其中待处理5条、处理中7条。"

## 注意事项

- 每次查询前先确认权限范围，不越权查询
- 涉及客户隐私字段，按业务规范脱敏展示
- 不要在没有确认参数的情况下擅自扩大查询范围（比如用户只问"我的"，不要查全公司的）
- 接口字段和参数如与 `references/api.md` 记录不一致，以接口实际返回为准，并提醒维护者更新参考文档
