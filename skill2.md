# Ticket Skill

## 作用

该 Skill 提供工单相关业务能力。

当用户希望查询、管理、处理工单时，应优先使用本 Skill，而不是直接回答。

所有业务数据必须通过 API 获取，不允许根据历史上下文推断。

---

## 适用场景

当用户存在以下意图时，应使用本 Skill：

- 查询工单
- 查看某个工单详情
- 查询工单状态
- 查询待处理工单
- 查询我创建的工单
- 查询某个客户的工单
- 分配工单负责人
- 导出工单
- 统计工单数量
- 查询最近创建的工单

例如：

> 我的待处理工单有哪些？

> 工单 T20240612001 当前状态是什么？

> 把工单 T20240612001 分配给张三。

> 导出最近一个月所有已完成工单。

---

## 能力列表

本 Skill 提供以下 API：

| API | 用途 |
|------|------|
| query_ticket_list | 查询工单列表 |
| query_ticket_detail | 查询工单详情 |
| query_ticket_status | 查询工单状态 |
| assign_ticket | 分配工单责任人 |
| export_ticket | 导出工单 |

对应 API 文档位于 references 目录。

---

## 调用原则

### 查询类

优先调用：

- query_ticket_list
- query_ticket_detail
- query_ticket_status

不要猜测工单信息。

---

### 修改类

涉及修改数据（例如分配负责人）时：

先确认目标工单存在。

再调用 assign_ticket。

如果接口返回失败，应将错误信息反馈给用户。

---

### 导出类

导出操作必须调用 export_ticket。

不要生成虚假的 Excel 或 CSV。

---

## 参数补全

如果用户未提供完整条件，应根据上下文补全。

例如：

用户：

> 查看昨天创建的工单

应自动解析：

创建时间 = 昨天

如果缺少必要信息，应继续询问，例如：

> 导出哪些状态的工单？

---

## 注意事项

- 不允许虚构工单。
- 不允许缓存业务数据。
- 每次查询均调用 API。
- API 返回为空时，应明确说明没有查询到数据。
- 修改类操作必须依据 API 返回结果反馈用户。
